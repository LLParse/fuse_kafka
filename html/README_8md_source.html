6 
7 Intercepts all writes to specified directories and send them
8 to apache kafka brokers. Quite suited for log centralization.
9 
10 Installing
11 ==========
12 
13 Packages for various distros can be installed from [these repositories](http://download.opensuse.org/repositories/home:/yazgoo/) at [openSUSE Build Service](https://build.opensuse.org/package/show/home:yazgoo/fuse_kafka).
14 
15 The following should install the new repositories then install fuse\_kafka:
16 
17  # curl -O \
18  https://raw.githubusercontent.com/yazgoo/fuse_kafka/master/setup_packages.sh \
19  && md5sum -c <(echo "a9c81807fa1f84018f90bbd80e987863 setup_packages.sh") \
20  && chmod +x setup_packages.sh && ./setup_packages.sh

22 Configuration
23 =============
24 
25 A default configuration file is available in conf/fuse\_kafka.properties.
26 An explanation for each parameter is available in this file.
27 The packages should install it in /etc/fuse\_kafka.conf.
28 
29 Quickstart (from sources)
30 =========================
31 
32 If you want to test fuse\_kafka, using a clone of this repository.
33 
34 First, build it:
35 
36  $ ./build.py
37 
38 On another terminal session, start zookeeper (this will download kafka):
39 
40  $ ./build.py zookeeper_start
41 
42 On another one, start kafka:
43 
44  $ ./build.py kafka_start
45 
46 The default configuration is conf/fuse\_kafka.properties.
47 An important piece of configuration is fuse\_kafka\_directories:
48 
49  $ grep fuse_kafka_directories conf/fuse_kafka.properties -B2
50  # directories fuse_kafka will listen to (launch script will try to
51  # create them if they don't exist)
52  fuse_kafka_directories=["/tmp/fuse-kafka-test"]
53 
54 Start fuse\_kafka using the init script:
55 
56  $ src/fuse_kafka.py start
57 
58 If you're not running as root, you might have to make
59 /etc/fuse.conf readable by your user (here to all users):
60 
61  $ chmod a+r /etc/fuse.conf
62 
63 And allow non-root user to specify the allow\_other option, by adding
64 a line with user\_allow\_other in /etc/fuse.conf.
65 
66 If fuse\_kafka is running, you should get the following output when
67 running:
68 
69  $ src/fuse_kafka.py status
70  listening on /tmp/fuse-kafka-test
71  fuse kafka is running
72 
73 In yet another terminal, start a test consumer:
74 
75  $ ./build.py kafka_consumer_start
76 
77 Then start writing to a file under the overlay directory:
78 
79  $ bash -c 'echo "foo"' > /tmp/fuse-kafka-test/bar
80 
81 You should have an output from the consumer similar to this:
82 
83  event:
84  group: users
85  uid: 1497
86  @tags:
87  - test
88  @fields:
89  hostname: test
90  @timestamp: 2014-10-03T09:07:04.000+0000
91  pid: 6485
92  gid: 604
93  command: bash -c echo "foo"
94  @message: foo
95  path: /tmp/fuse-kafka-test/bar
96  @version: 0.1.3
97  user: yazgoo
98 
99 When you're done, you can stop fuse\_kafka:
100 
101  $ src/fuse_kafka.py stop
102 
103 Quota option test
104 =================
105 
106 First, commment fuse\_kafka\_quota in conf/fuse\_kafka.properties.
107 Then, start fuse kafka.
108 
109  $ src/fuse_kafka.py start
110 
111 Let's create a segfaulting program:
112 
113  $ cat first.c
114  int main(void)
115  {
116  *((int*)0) = 1;
117  }
118 
119  $ gcc first.c
120 
121 Then start a test consumer, displaying only the path and message\_size-added fields
122 
123 Launch the segfaulting program in fuse-kafka-test directory:
124 
125  $ /path/to/a.out
126 
127 A new core file should appear in fused directory.
128 
129 Here is the consumer output:
130 
131  $ SELECT="message_size-added path" ./build.py kafka_consumer_start
132  event:
133  message_size-added: 4096
134  path: /tmp/fuse-kafka-test/core
135  ...
136  event:
137  message_size-added: 4096
138  path: /tmp/fuse-kafka-test/core
139 
140 Here we see many messages.
141 
142 Then, uncomment fuse\_kafka\_quota in conf/fuse\_kafka.properties and
143 launch the segfaulting program,
144 
145  $ SELECT="message_size-added path" ./build.py kafka_consumer_start
146  event:
147  message_size-added: 64
148  path: /tmp/fuse-kafka-test/core
149 
150 This time, we only receive the first write.

152 Event format
153 ============
154 
155 We use a logstash event, except the message and command are base64 encoded:
156 
157  {"path": "/var/log/redis_6380.log", "pid": 1262, "uid": 0, "gid": 0,
158  "@message": "aGVsbG8gd29ybGQ=",
159  "@timestamp": "2014-09-11T14:19:09.000+0000","user": "root", "group":
160  "root",
161  "command": "L3Vzci9sb2NhbC9iaW4vcmVkaXMtc2VydmVyIC",
162  "@version": "0.1.2",
163  "@fields": {
164  "first_field": "first_value",
165  "second_field": "second_value" },
166  "@tags": ["mytag"]}
